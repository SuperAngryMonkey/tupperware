# Tupperware v0.1.4 Release Notes

**The "actually deployable" release.** All changes focus on making Tupperware work on Proxmox hosts that aren't identical to the developer's. If you've ever tried to install Tupperware on a host with non-default storage, a different bridge name, or unclear Tailscale setup state, this release is for you.

## New features

### `tupperware-preflight` — validate before you install

A new command that runs every check needed for a successful install **before** you've changed anything on the host. Catches:

- Proxmox VE detection and version
- Internet connectivity and DNS
- Bridge availability (defaults to `vmbr0`, lists alternatives if not found)
- Storage backend availability (defaults to `local-lvm`, lists alternatives)
- Disk space on /
- Debian 12 template downloaded
- VMID 9000 not already in use
- Tailscale CLI installed and authenticated on host
- OAuth credentials file present, readable, properly permissioned, and valid
- OAuth client authorized for `tag:lxc` (mints a test key to verify)

Output is color-coded:
- Green checkmarks for things that pass
- Yellow warnings for things that could trip you up (e.g., bridge `vmbr0` doesn't exist on this host)
- Red errors for things that will definitely fail (e.g., OAuth credentials missing)

Exit codes:
- `0` = ready
- `1` = warnings only (install will likely work)
- `2` = errors (install will fail)

Usage:
```bash
tupperware-preflight
```

### Bridge selection in template builder

`tupperware-build-template` now supports `--bridge <name>` flag and `BRIDGE=<name>` env var. Default remains `vmbr0`. Validates the bridge exists before creating the container.

```bash
# Default bridge
tupperware-build-template

# Custom bridge
tupperware-build-template --bridge vmbr1
BRIDGE=vmbr1 tupperware-build-template
```

If the bridge doesn't exist, fails fast and lists what's available.

### `tupperware-uninstall` — clean removal

Reversible install means you can experiment without committing. Removes:
- Web UI service and files
- CLI scripts (including legacy names from older versions)
- Optionally: the LXC template (asks first)
- Optionally: OAuth credentials (asks first)

Does **not** remove cloned containers — those stay running.

```bash
tupperware-uninstall
```

### Dynamic Debian template lookup

`tupperware-build-template` now auto-detects the latest available `debian-12-standard` template via `pveam available`. No more hardcoded patch versions in scripts that go stale.

## Recommended deployment flow for new hosts

```bash
# 1. Stash OAuth credentials
mkdir -p /root/.tailscale && chmod 700 /root/.tailscale
cat > /root/.tailscale/oauth <<'EOF'
TS_OAUTH_CLIENT_ID=...
TS_OAUTH_CLIENT_SECRET=...
EOF
chmod 600 /root/.tailscale/oauth

# 2. Install Tupperware tooling
curl -fsSL https://raw.githubusercontent.com/SuperAngryMonkey/tupperware/main/scripts/install.sh | bash

# 3. Run preflight check
tupperware-preflight
# Fix any errors/warnings before continuing.

# 4. Build the template
tupperware-build-template
# Or with overrides:
# tupperware-build-template --bridge vmbr1 --storage local-zfs

# 5. Install the web UI
curl -fsSL https://raw.githubusercontent.com/SuperAngryMonkey/tupperware/main/scripts/install-webui.sh | bash

# 6. Open http://<this-host>:8080/
```

If something goes wrong, `tupperware-uninstall` to reset, fix the issue, run preflight again.

## Upgrading from v0.1.3

Same as before:

```bash
curl -fsSL https://raw.githubusercontent.com/SuperAngryMonkey/tupperware/main/scripts/install.sh | bash
curl -fsSL https://raw.githubusercontent.com/SuperAngryMonkey/tupperware/main/scripts/install-webui.sh | bash
```

This will also install the new `tupperware-preflight` and `tupperware-uninstall` commands.
