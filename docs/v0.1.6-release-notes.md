# Tupperware v0.1.6 Release Notes

**Security hardening release.** No new functionality — just clearer guardrails around OAuth credentials so people don't accidentally use someone else's keys.

## What changed

### Prominent OAuth security warning in README

Added a callout box at the top of the README, before any install instructions. Says clearly:
- OAuth credentials are unique to your tailnet
- Never copy from a guide, screenshot, or someone else's install
- Always generate your own via the Tailscale admin console
- Rotation steps if anything seems off

### `tupperware-preflight` detects placeholder values

If your `/root/.tailscale/oauth` contains placeholder strings like `PASTE_YOUR_CLIENT_ID_HERE`, `YOUR_CLIENT_SECRET_HERE`, or `<...>`, preflight now fails fast with a clear message pointing you to the Tailscale admin console:

```
[✗] FAIL: OAuth file contains placeholder values, not real credentials
    You appear to have copied the README template literally.
    Generate YOUR OWN OAuth client in the Tailscale admin console:
      https://login.tailscale.com/admin/settings/oauth
    See docs/security.md for why these credentials are unique-per-tailnet.
```

Previously, the install would succeed silently and fail later at clone time with a confusing OAuth error.

### New `docs/security.md`

A dedicated security doc covering:
- Threat model for the OAuth client (what it can/cannot do)
- Blast radius of a leaked credential
- Step-by-step credential rotation procedure
- FAQ about shared vs. per-host clients
- Recommendations for tailnet ACL hygiene

Linked from the README warning.

## Upgrading from v0.1.5

```bash
curl -fsSL https://raw.githubusercontent.com/SuperAngryMonkey/tupperware/main/scripts/install.sh | bash
```

The new preflight check runs whenever you invoke `tupperware-preflight` — no service changes needed. Existing valid credentials continue to work without modification.

## Bug fixes

None — this is a documentation and validation release.
