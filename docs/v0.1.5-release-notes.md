# Tupperware v0.1.5 Release Notes

**Template decoupled from install.** Tupperware no longer builds the template as part of its core install — the template is now a prerequisite the user provides, like OAuth credentials. The web UI shows a clear setup page when no template is found, with three install paths.

## Why this changes

Before v0.1.5, installing Tupperware on a new host meant 4 minutes of waiting for the template build to finish before you could even see the UI. If you already had a Debian 12 LXC you liked, that was wasted work.

v0.1.5 turns "build the template" into one of three options, presented in the UI itself.

## What's new

### Web UI: friendly setup page when no template exists

If the web UI starts and there's no template at VMID 9000 (configurable), instead of crashing or silently failing, you see a clean setup page with three options:

1. **Download sample template** — `tupperware-import-template` pulls a pre-built `.tar.zst` from GitHub Releases and restores it (~2 minutes)
2. **Build from scratch** — `tupperware-build-template` does the full Debian 12 + Tailscale install (~4 minutes)
3. **Use your own template** — point Tupperware at a different VMID via `TEMPLATE_VMID` env var

Once a template exists, refresh and the normal UI appears.

### New CLI: `tupperware-import-template`

Downloads a pre-built Tupperware-ready Debian 12 template from GitHub Releases and restores it as a Proxmox template at the configured VMID.

```bash
# Default — VMID 9000, local-lvm storage
tupperware-import-template

# Custom VMID and storage
tupperware-import-template --vmid 9001 --storage local-zfs

# Custom URL (e.g., your own template hosted elsewhere)
tupperware-import-template --url https://example.com/my-template.tar.zst
```

### New CLI: `tupperware-export-template`

For users who build their own template (with custom packages, configs, etc.) and want to share it. Exports an existing template to a `.tar.zst` file suitable for uploading to GitHub Releases or distributing to other hosts.

```bash
# Export VMID 9000 to ./tupperware-template.tar.zst
tupperware-export-template

# Custom output
tupperware-export-template --vmid 9001 --output /tmp/my-template.tar.zst
```

### Pre-built template hosted on GitHub Releases

The v0.1.5 release ships with `tupperware-template.tar.zst` attached — a ~150MB pre-built Tupperware-ready Debian 12 LXC template. `tupperware-import-template` downloads this by default.

## Recommended new-host workflow

```bash
# 1. Stash OAuth credentials
mkdir -p /root/.tailscale && chmod 700 /root/.tailscale
cat > /root/.tailscale/oauth <<'EOF'
TS_OAUTH_CLIENT_ID=...
TS_OAUTH_CLIENT_SECRET=...
EOF
chmod 600 /root/.tailscale/oauth

# 2. Install Tupperware CLI tools
curl -fsSL https://raw.githubusercontent.com/SuperAngryMonkey/tupperware/main/scripts/install.sh | bash

# 3. Install the web UI
curl -fsSL https://raw.githubusercontent.com/SuperAngryMonkey/tupperware/main/scripts/install-webui.sh | bash

# 4. Open http://<host>:8080/ -> shows setup page
# 5. Run ONE of:
tupperware-import-template      # fastest
tupperware-build-template       # full build

# 6. Refresh browser -> ready to clone
```

## Upgrading from v0.1.3 or v0.1.4

```bash
curl -fsSL https://raw.githubusercontent.com/SuperAngryMonkey/tupperware/main/scripts/install.sh | bash
curl -fsSL https://raw.githubusercontent.com/SuperAngryMonkey/tupperware/main/scripts/install-webui.sh | bash
```

Existing templates continue to work — Tupperware just checks for existence at VMID 9000 (or whatever you set `TEMPLATE_VMID` to). If your template is there, nothing changes.
