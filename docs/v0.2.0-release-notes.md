# Tupperware v0.2.0 Release Notes

**Host-to-host container transfer.** Move LXC containers between Proxmox hosts over Tailscale with a single click in the web UI, or one CLI command.

## What's new

### Transfer button on every container in the inventory

The inventory panel in the web UI now has a **TRANSFER** button on every container row. Clicking it opens a modal where you pick:

- **Destination host**: dropdown auto-populated with other `tag:prox-host` Proxmox hosts on your tailnet
- **Destination storage**: dropdown auto-populated by SSH'ing the destination and querying `pvesm status`
- **Identity mode**: FRESH (new tailnet IP) or PRESERVE (same machine key)

Output streams live to the provisioning console.

### `tupperware-transfer` CLI

For scripting / cron / CI use:

```bash
tupperware-transfer <src-vmid> <dest-fqdn> [--storage <name>] [--preserve-identity]
```

### `tupperware-rejoin` helper

Runs on the destination host to mint a fresh auth key and inject it into a restored container. Called automatically by `tupperware-transfer` in fresh-identity mode. Can be re-run manually if Tailscale fails to come up after a transfer.

### Transfer history panel

The web UI now shows the last 10 transfers (from `/var/log/tupperware/transfer.log`), including source/destination, identity mode, transfer size, duration, and status.

### Dynamic host discovery via Tailscale API

The transfer modal's destination dropdown is populated by querying the Tailscale Devices API for hosts tagged `tag:prox-host`. No static config needed — tag your hosts and they appear automatically.

## How a transfer works

Cold migration: stop source → vzdump → scp over Tailscale → pct restore on destination → optional rejoin → start. The source container is left stopped (NOT destroyed) so you can verify before manually destroying.

See [docs/v0.2-transfer.md](docs/v0.2-transfer.md) for the full guide.

## Prerequisites

- Both Proxmox hosts running Tupperware v0.2+
- Both hosts tagged `tag:prox-host` in your tailnet
- ACL grant allowing SSH between `tag:prox-host` machines
- OAuth client has `Devices: Read` scope (for dynamic host discovery)

Full setup steps in `docs/v0.2-transfer.md`.

## Files added/changed

- `scripts/tupperware-transfer.sh` — new, the transfer workhorse
- `scripts/tupperware-rejoin.sh` — new, mints auth keys on destination
- `webui/app.py` — transfer modal, history panel, prox-host discovery API
- `scripts/install.sh` — installs new scripts, creates log directory, sets up logrotate
- `docs/v0.2-transfer.md` — new, usage guide

## Audit log format

Each transfer writes a JSON line to `/var/log/tupperware/transfer.log`:

```json
{"ts":"2026-06-17T18:30:00Z","src_vmid":200,"src_hostname":"lab-lxc-01","dest_host":"proxmox.bobcat-gondola.ts.net","dest_vmid":115,"identity":"fresh","storage":"local-lvm","size_bytes":192937472,"duration_seconds":42,"status":"success"}
```

Logrotate config rotates monthly, keeps 12 months, compresses old logs.

## Upgrading from v0.1.6

```bash
curl -fsSL https://raw.githubusercontent.com/SuperAngryMonkey/tupperware/main/scripts/install.sh | bash
curl -fsSL https://raw.githubusercontent.com/SuperAngryMonkey/tupperware/main/scripts/install-webui.sh | bash
```

Then complete the tagging + ACL steps in `docs/v0.2-transfer.md`.

## Known limitations

- Cold migration only (container is stopped during transfer)
- No bandwidth limit (deferred to v0.3)
- Source not auto-destroyed (intentional — you verify first)
- No rollback if `pct restore` fails on destination
