# Tupperware v0.2 — Container Transfer Between Proxmox Hosts

v0.2 adds the ability to move LXC containers between Proxmox hosts over Tailscale, with a single click in the web UI or one command from the CLI.

## Prerequisites

### Both Proxmox hosts must be on the same tailnet

If you have two Proxmox hosts running Tupperware, both must be authenticated to the same Tailscale tailnet. Test connectivity from the source:

```bash
ssh -o ConnectTimeout=5 root@<dest-tailnet-fqdn> whoami
```

Should return `root` without prompting for a password.

### Tag both hosts `tag:prox-host`

In the Tailscale admin console (https://login.tailscale.com/admin/machines):

1. Click each Proxmox host
2. Edit tags → add `tag:prox-host`
3. Save

Then add the tag to your ACL policy. Open https://login.tailscale.com/admin/acls/file and ensure these blocks exist:

```hujson
"tagOwners": {
    "tag:prox-host": ["autogroup:owner"],
    // ... other tags ...
},

"grants": [
    // ... your existing grants ...

    // Allow Tupperware hosts to SSH each other for transfers
    {
        "src": ["tag:prox-host"],
        "dst": ["tag:prox-host"],
        "ip":  ["*"]
    }
],

"ssh": [
    // ... existing ssh rules ...

    {
        "src":    ["tag:prox-host"],
        "dst":    ["tag:prox-host"],
        "users":  ["root"],
        "action": "accept"
    }
]
```

### OAuth client needs Devices: Read scope

For dynamic destination host discovery in the web UI, your OAuth client needs the `Devices: Read` scope in addition to `Auth Keys: Write`.

Go to https://login.tailscale.com/admin/settings/oauth, edit your existing Tupperware OAuth client, and add the `Devices: Read` scope. The existing client ID/secret continue to work — no need to regenerate or update `/root/.tailscale/oauth`.

## How a transfer works

1. **Source side**: Tupperware stops the source LXC and runs `vzdump` to create a `.tar.zst` archive
2. **Transport**: The archive is `scp`'d to the destination over Tailscale (`tailscale0` interface)
3. **Destination side**: `pct restore` extracts the archive on the chosen storage backend, assigning the next free VMID
4. **Identity**: Depending on mode:
   - **Fresh (default)**: Tailscale state is wiped from the source before vzdump, and `tupperware-rejoin` mints a fresh auth key on the destination so the container appears as a new device on the tailnet
   - **Preserve**: The Tailscale machine key is preserved, so the container keeps the same tailnet IP. The source's `onboot=0` is set after to prevent accidental dual-boot conflicts
5. **Start**: The destination container starts and joins the tailnet
6. **Audit**: A JSON entry is appended to `/var/log/tupperware/transfer.log`
7. **Source preserved**: The source container is left stopped on the source host — NOT destroyed. You verify the destination, then manually `pct destroy <vmid> --purge` when ready.

## Using the web UI

1. Open `http://<source-host>:8080/`
2. Find the container in the INVENTORY table
3. Click the **TRANSFER** button on its row
4. In the modal:
   - **DESTINATION HOST**: pick from your other `tag:prox-host` machines (auto-discovered)
   - **DESTINATION STORAGE**: pick a storage backend on the destination (auto-discovered via SSH)
   - **IDENTITY**: FRESH (default, new tailnet IP) or PRESERVE (same machine key)
5. Click **TRANSFER**

Output streams live to the provisioning console.

## Using the CLI

```bash
# Default — fresh identity, local-lvm on destination
tupperware-transfer 200 proxmox.bobcat-gondola.ts.net

# Specify destination storage
tupperware-transfer 200 proxmox.bobcat-gondola.ts.net --storage local-zfs

# Preserve identity (advanced)
tupperware-transfer 200 proxmox.bobcat-gondola.ts.net --preserve-identity
```

## Verifying a transfer

After a transfer completes:

```bash
# Confirm container started on destination
ssh root@<dest-fqdn> pct list

# Confirm it joined the tailnet
ssh root@<dest-fqdn> "pct exec <dest-vmid> -- tailscale status | head -3"

# Then when satisfied, destroy the source
pct destroy <src-vmid> --purge
```

## Audit log

Every transfer (success or failure) is logged to `/var/log/tupperware/transfer.log` as JSON, one entry per line. Example:

```json
{"ts":"2026-06-17T18:30:00Z","src_vmid":200,"src_hostname":"lab-lxc-01","dest_host":"proxmox.bobcat-gondola.ts.net","dest_vmid":115,"identity":"fresh","storage":"local-lvm","size_bytes":192937472,"duration_seconds":42,"status":"success"}
```

The web UI shows the last 10 entries in the TRANSFER HISTORY panel.

Log rotation is set up via `/etc/logrotate.d/tupperware` (monthly, 12 rotations, compressed).

## Limitations / known caveats

- **Cold migration only**: containers are stopped during transfer. Live migration would require shared storage or block-level streaming — not in scope for v0.2.
- **No bandwidth limit**: transfer uses the full Tailscale tunnel bandwidth. Deferred to v0.3.
- **Source not auto-destroyed**: by design — you verify before destroying.
- **No rollback**: if `pct restore` fails on destination, the source container is left stopped. Restart it manually with `pct start <vmid>` if you want it back up.
- **Storage must exist on destination**: Tupperware validates this before starting; will fail early with a clear message if the chosen storage isn't on the destination.

## Troubleshooting

**"Cannot SSH to root@<dest-fqdn>"**
- Verify both hosts are tagged `tag:prox-host`
- Verify ACL has SSH grant `tag:prox-host → tag:prox-host`
- Verify Proxmox firewall isn't blocking tailscale0
- Test manually: `ssh root@<dest-fqdn> whoami`

**"Destination does not have Tupperware installed"**
- Run the install script on the destination: `curl -fsSL https://raw.githubusercontent.com/SuperAngryMonkey/tupperware/main/scripts/install.sh | bash`

**"Storage X not available on destination"**
- Check what's available: `ssh root@<dest-fqdn> "pvesm status -content rootdir"`
- Pick a storage that supports container content

**Transfer succeeded but container has no tailnet IP**
- Re-run rejoin manually: `ssh root@<dest-fqdn> tupperware-rejoin <dest-vmid>`
- Check OAuth credentials on destination: `ssh root@<dest-fqdn> ls -la /root/.tailscale/oauth`
- Check OAuth has `Auth Keys: Write` scope with `tag:lxc` authorized
