# Tupperware v0.2 — Container Transfer

**Status:** Design / planning. Not yet implemented.
**Last updated:** May 2026.

---

## Goal

Extend Tupperware to move LXC containers between Proxmox hosts that don't share a LAN but can see each other over Tailscale. One Tupperware UI initiates a transfer; the destination Tupperware (passively) receives.

Concrete scenario: Ghost runs Tupperware on `proxlab-fl` (Lauderdale) and `proxlab-gib` (Gibraltar). The two hosts can't ping each other on LAN, but both are on the same tailnet. From the FL UI, he wants to select an LXC, pick "send to proxlab-gib," and have it appear running on the Gibraltar host a few minutes later.

---

## Decisions (locked)

| Question | Decision |
|---|---|
| Push or pull? | **Push** — source initiates |
| Live or cold? | **Cold migration** — `pct stop`, transfer, `pct start` |
| Transport? | **SSH/SCP over Tailscale** — leverage existing Tailscale SSH grants, no new keys to manage |
| Tailscale identity after move? | **Per-transfer choice in UI** — default fresh, "preserve identity" as advanced option |
| Auto-destroy source after success? | **No, never** — user verifies and destroys manually. Tupperware is non-destructive by default. |
| Host discovery? | **Dynamic via Tailscale API** — query for `tag:prox-host` devices. No static config file. |
| Bandwidth limits? | **Out of scope for v0.2** — defer to v0.3 if requested |
| Transfer audit log? | **Yes** — `/var/log/tupperware/transfer.log` |
| Destination storage backend? | **Selectable per-transfer** — dropdown populated by querying destination's `pvesm status` |

## High-level flow

```
SOURCE HOST (proxlab-fl)                DEST HOST (proxlab-gib)
┌────────────────────────┐              ┌────────────────────────┐
│  Tupperware UI         │              │  Tupperware UI         │
│  ┌──────────────────┐  │              │  (passive — no special │
│  │ NEW    TRANSFER  │  │              │   endpoint, just SSH'd │
│  └──────────────────┘  │              │   into by source)      │
│  [Inventory]           │              │                        │
│   200 webserver-01 ●   │              │  [Inventory]           │
│   201 db-cache    ●    │              │   200 webserver-01 ●   │
│  [Send to host]        │              │   (after transfer)     │
│   ▼ proxlab-gib        │              │                        │
│  [Storage on dest]     │              │                        │
│   ▼ local-lvm          │              │                        │
│  [Identity: fresh]     │              │                        │
│  [Package & Send]      │              │                        │
└──────────┬─────────────┘              └────────────────────────┘
           │
           │ 1. pct stop <vmid>
           │ 2. vzdump → /tmp/dump.tar.zst (with progress streamed to UI)
           │ 3. scp dump.tar.zst root@<dest>:/var/lib/vz/dump/
           │ 4. ssh <dest> "pct restore <newvmid> dump --storage <chosen-storage>"
           │ 5. (if fresh identity) ssh <dest> "tupperware-rejoin <newvmid>"
           │ 6. ssh <dest> "pct start <newvmid>"
           │ 7. cleanup local dump file
           │ 8. write JSON record to /var/log/tupperware/transfer.log
           │
           ▼
       Container exists on proxlab-gib with fresh local VMID.
       Source container remains stopped on proxlab-fl until manually destroyed.
```

The destination host's Tupperware doesn't expose any new HTTP endpoints. It just shows the new container in its inventory like any other, because the container appeared via standard Proxmox commands.

## Authentication: Tailscale SSH between hosts

We piggyback on Tailscale SSH so we don't have to manage SSH keys manually.

### Required tailnet ACL additions

A new tag for Proxmox hosts and an SSH grant so they can SSH each other:

```hujson
"tagOwners": {
  "tag:lxc": ["autogroup:admin"],
  "tag:prox-host": ["autogroup:admin"],
},

"ssh": [
  // Existing: users can SSH to their containers
  {
    "action": "accept",
    "src":    ["autogroup:member"],
    "dst":    ["tag:lxc"],
    "users":  ["root", "autogroup:nonroot"],
  },
  // NEW: Proxmox hosts can SSH each other for transfers
  {
    "action": "accept",
    "src":    ["tag:prox-host"],
    "dst":    ["tag:prox-host"],
    "users":  ["root"],
  },
],
```

Each Proxmox host must be tagged `tag:prox-host` when it joins Tailscale (either at `tailscale up` time, or via the admin console after the fact).

## Host discovery — dynamic via Tailscale API

**No static config file.** Tupperware queries the Tailscale API for all devices currently tagged `tag:prox-host` and uses that as the destination dropdown.

### Implementation

Reuse the existing OAuth client (which already mints auth keys for `tag:lxc`). Add an additional scope: **Devices: Read** so Tupperware can enumerate tagged devices.

Flow on every UI load of the Transfer tab:

```python
def list_prox_hosts():
    token = get_oauth_token()  # cached ~50min
    resp = requests.get(
        "https://api.tailscale.com/api/v2/tailnet/-/devices",
        headers={"Authorization": f"Bearer {token}"}
    )
    self_hostname = subprocess.check_output(["hostname"], text=True).strip()
    hosts = []
    for dev in resp.json().get("devices", []):
        if "tag:prox-host" not in dev.get("tags", []):
            continue
        if dev.get("hostname") == self_hostname:
            continue  # exclude self
        hosts.append({
            "name": dev.get("hostname"),
            "fqdn": dev.get("name"),  # full tailnet FQDN
            "ip": dev.get("addresses", [None])[0],
            "online": is_recent(dev.get("lastSeen")),
        })
    return hosts
```

Result: the Transfer tab always shows the current set of Proxmox hosts on your tailnet. Add a new host to the tailnet, tag it `prox-host`, and it shows up immediately in the dropdown on every other Tupperware host. No config file edits, no service restarts.

### OAuth scope expansion

| v0.1 scopes | v0.2 scopes |
|---|---|
| Auth Keys: Write | Auth Keys: Write |
| | **Devices: Read** (new) |

Existing v0.1 users edit their OAuth client to add `Devices: Read`. README documents this in the v0.2 prereqs.

### Caching

Tailscale API call is cheap but not free. Cache the device list for 60 seconds in Tupperware. UI shows a small "refresh" button that bypasses cache.

## Destination storage backend selection

Each Tupperware host knows its own available LXC storage backends. When the user picks a destination host in the Transfer dialog, Tupperware queries that host (over SSH) for its storage list:

```bash
ssh root@<dest-fqdn> "pvesm status -content rootdir | tail -n +2 | awk '{print \$1}'"
```

Returns something like:
```
local-lvm
local-zfs
backup-nfs
```

The UI shows a "Storage on destination" dropdown populated with this list. Default selection is `local-lvm` (or whatever the first one is). User can pick.

The selected storage gets passed to `pct restore --storage <storage>` on the destination.

## Transfer audit log

Every transfer attempt is logged to `/var/log/tupperware/transfer.log`. Format: one JSON line per transfer.

```json
{"ts":"2026-05-05T22:13:45Z","src_vmid":200,"src_hostname":"webserver-01","dest_host":"proxlab-gib","dest_vmid":207,"identity":"fresh","storage":"local-lvm","size_bytes":3221225472,"duration_seconds":87,"status":"success"}
{"ts":"2026-05-05T22:18:02Z","src_vmid":201,"src_hostname":"db-cache","dest_host":"proxlab-bahamas","dest_vmid":null,"identity":"fresh","storage":"local-lvm","size_bytes":null,"duration_seconds":12,"status":"failed","error":"scp: connection timeout"}
```

The web UI has a "Transfer history" panel below the inventory that tails the last 10 entries from this log, color-coded by status. Failed transfers show the error inline; clicking shows full details.

Logrotate config dropped at `/etc/logrotate.d/tupperware`:
```
/var/log/tupperware/transfer.log {
    monthly
    rotate 12
    compress
    missingok
    notifempty
}
```

## Tailscale identity handling

When an LXC moves between hosts, what happens to its Tailscale identity?

### Option A: Fresh identity (default)

Before vzdump on source, wipe Tailscale state inside the container:
```bash
pct exec <vmid> -- bash -c '
  tailscale logout
  systemctl stop tailscaled
  rm -rf /var/lib/tailscale/*
'
```

Also re-arm the firstboot service so it'll join again on the destination:
```bash
pct exec <vmid> -- systemctl enable tailscale-firstboot.service
```

After restore on destination, Tupperware mints a new auth key and writes it to `/etc/tailscale/authkey`. On `pct start`, the firstboot service runs and the container joins the tailnet as a new device.

**User experience:**
- Tailnet IP changes (e.g., 100.10.20.30 → 100.40.50.60)
- Old entry shows as offline in admin console (eventually expires after 180d default)
- Hostname stays the same (Tailscale handles uniqueness — adds suffix if collision)

### Option B: Preserve identity (advanced)

Don't wipe Tailscale state. The vzdump captures `/var/lib/tailscale/tailscaled.state`, which contains the device's machine key. After restore, the container has the same identity.

**User experience:**
- Same tailnet IP
- Same admin console entry (just shows new direct/derp endpoints)
- No re-auth needed

**Risks:**
- If you accidentally start the source container after starting the destination, you have two nodes claiming the same machine key. Tailscale will refuse the second one (good) but it's confusing.
- If you ever restart the source for any reason post-transfer, it'll fight with the destination.

Mitigation: after a successful "preserve identity" transfer, Tupperware sets `pct set <vmid> -onboot 0` on the source to prevent accidental autostart. User still has to manually destroy.

### Implementation note

The choice is exposed as a radio button in the transfer dialog. Default fresh; preserve identity is the second option with a tooltip explaining the constraint.

## CLI tool

A new CLI command, used by the web UI but also runnable directly:

```bash
tupperware-transfer <local-vmid> <dest-fqdn> [--preserve-identity] [--storage <name>]

# Examples
tupperware-transfer 200 proxlab-gib.<your-tailnet>.ts.net
tupperware-transfer 201 proxlab-bahamas.<your-tailnet>.ts.net --preserve-identity --storage local-zfs
```

`<dest-fqdn>` is the full Tailscale FQDN of the destination host (resolvable via Tailscale DNS / MagicDNS). The script doesn't need a config file — the user (or web UI) provides the FQDN directly.

If `--storage` is omitted, defaults to `local-lvm` on the destination.

Output is line-streamed (the web UI captures via Popen and streams via SSE).

After completion, appends a JSON line to `/var/log/tupperware/transfer.log`.

## Error handling and recovery

### Source-side failures

| Failure point | Recovery |
|---|---|
| `pct stop` fails | Abort, container remains running, error to user |
| `vzdump` fails | Restart container (`pct start`), clean up partial file in `/tmp`, error to user |
| Local disk full during dump | Same as above |

### Network/transfer failures

| Failure point | Recovery |
|---|---|
| `scp` fails (network blip) | Restart source container, delete partial file on dest via SSH, error to user. **Important:** always clean up partial dump on dest, otherwise future restores might pick the wrong file. |
| Dest disk full | Same |

### Destination-side failures

| Failure point | Recovery |
|---|---|
| `pct restore` fails | Restart source container, clean up dump on dest, error to user |
| `pct start` on dest fails | Source remains stopped, dest container exists but won't start. Error to user with both VMIDs and instructions. |
| Tailscale firstboot fails on dest | Container is up, just not on tailnet. User can re-trigger via `pct exec <vmid> -- systemctl start tailscale-firstboot.service` after fixing the underlying issue. |

### General principle

**Never destroy the source container automatically.** If anything goes wrong after the source is stopped, the worst case is "user has to `pct start` it again." Data is never lost.

If SCP completes but anything after fails, the dump file is preserved on both sides for manual recovery.

All failures are logged to `/var/log/tupperware/transfer.log` with `status="failed"` and the error message.

## Pre-flight checks

The UI runs these before enabling the "Start Transfer" button:

| Check | If fail |
|---|---|
| Source container exists | "VMID X not found" |
| Source is not a template | "VMID X is a template, can't transfer templates" |
| Destination is reachable via Tailscale | "Can't reach proxlab-gib (offline?)" |
| SSH to destination works | "Can't SSH to proxlab-gib (Tailscale ACL allows tag:prox-host → tag:prox-host?)" |
| Source local disk has 2x container size free | "Need 8 GB free for dump, only 3 GB available" |
| Destination has Tupperware installed | "tupperware-* tools not found on dest" |
| Destination storage exists and has space | "Storage local-lvm full or not configured on dest" |
| Destination has free VMID | "No free VMIDs on destination" |

UI shows these as a checklist with green/red lights; "Start Transfer" stays disabled until all pass.

## Out of scope for v0.2

- **Live migration** (memory state moves with running container)
- **Multi-host orchestration** (move N containers in one operation)
- **Bandwidth limits** — defer to v0.3
- **Authentication/authz on the web UI** (still tailnet-only access controls)
- **Web UI on destination shows incoming transfers in real time** — would need pub/sub or polling between Tupperware instances
- **Cross-version compatibility** (transferring between PVE 8 and PVE 9 hosts) — both hosts must be same major version
