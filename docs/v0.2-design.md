# Tupperware v0.2 — Container Transfer

**Status:** Design / planning. Not yet implemented.
**Last updated:** May 2026.

---

## Goal

Extend Tupperware to move LXC containers between Proxmox hosts that don't share a LAN but can see each other over Tailscale. One Tupperware UI initiates a transfer; the destination Tupperware (passively) receives.

Concrete scenario: Ghost runs Tupperware on `proxlab-fl` (Lauderdale) and `proxlab-gib` (Gibraltar). The two hosts can't ping each other on LAN, but both are on the same tailnet. From the FL UI, he wants to select an LXC, pick "send to proxlab-gib," and have it appear running on the Gibraltar host a few minutes later.

---

## Decisions (locked)

| Question | Decision |
|---|---|
| Push or pull? | **Push** — source initiates |
| Live or cold? | **Cold migration** — `pct stop`, transfer, `pct start` |
| Transport? | **SSH/SCP over Tailscale** — leverage existing Tailscale SSH grants, no new keys to manage |
| Tailscale identity after move? | **Per-transfer choice in UI** — default fresh, "preserve identity" as advanced option |
| Auto-destroy source after success? | **No, never** — user verifies and destroys manually. Tupperware is non-destructive by default. |

## High-level flow

```
SOURCE HOST (proxlab-fl)                DEST HOST (proxlab-gib)
┌────────────────────────┐              ┌────────────────────────┐
│  Tupperware UI         │              │  Tupperware UI         │
│  ┌──────────────────┐  │              │  (passive — no special │
│  │ NEW    TRANSFER  │  │              │   endpoint, just SSH'd │
│  └──────────────────┘  │              │   into by source)      │
│  [Inventory]           │              │                        │
│   200 webserver-01 ●   │              │  [Inventory]           │
│   201 db-cache    ●    │              │   200 webserver-01 ●   │
│  [Send to host]        │              │   (after transfer)     │
│   ▼ proxlab-gib        │              │                        │
│   ▼ proxlab-bahamas    │              │                        │
│  [Package & Send]      │              │                        │
└──────────┬─────────────┘              └────────────────────────┘
           │
           │ 1. pct stop <vmid>
           │ 2. vzdump → /tmp/dump.tar.zst (with progress streamed to UI)
           │ 3. scp dump.tar.zst root@<dest>:/var/lib/vz/dump/ (with progress)
           │ 4. ssh <dest> "pct restore <newvmid> /var/lib/vz/dump/dump.tar.zst"
           │ 5. (if fresh identity) ssh <dest> "tupperware-rejoin <newvmid>"
           │ 6. ssh <dest> "pct start <newvmid>"
           │ 7. cleanup local dump file
           │
           ▼
       Container exists on proxlab-gib with fresh local VMID.
       Source container remains stopped on proxlab-fl until manually destroyed.
```

The destination host's Tupperware doesn't expose any new HTTP endpoints. It just shows the new container in its inventory like any other, because the container appeared via standard Proxmox commands.

## Authentication: Tailscale SSH between hosts

We piggyback on Tailscale SSH so we don't have to manage SSH keys manually.

### Required tailnet ACL additions

A new tag for Proxmox hosts and an SSH grant so they can SSH each other:

```hujson
"tagOwners": {
  "tag:lxc": ["autogroup:admin"],
  "tag:proxmox-host": ["autogroup:admin"],
},

"ssh": [
  // Existing: users can SSH to their containers
  {
    "action": "accept",
    "src":    ["autogroup:member"],
    "dst":    ["tag:lxc"],
    "users":  ["root", "autogroup:nonroot"],
  },
  // NEW: Proxmox hosts can SSH each other for transfers
  {
    "action": "accept",
    "src":    ["tag:proxmox-host"],
    "dst":    ["tag:proxmox-host"],
    "users":  ["root"],
  },
],
```

Each Proxmox host must be tagged `tag:proxmox-host` when it joins Tailscale. Either tag at `tailscale up` time, or via the admin console after the fact.

### OAuth scope expansion

The existing OAuth client is restricted to `tag:lxc`. For the host self-tagging to work via OAuth, we'd need either:
- A separate OAuth client for host onboarding (scoped to `tag:proxmox-host`), OR
- Tag hosts manually in the admin console (one-time, no scripting needed)

For v0.2, **manual tagging in admin console is fine**. Host onboarding isn't frequent.

## Host peer configuration

For the "Destination host" dropdown, Tupperware reads `/etc/tupperware/peers.conf`:

```
# /etc/tupperware/peers.conf
# Format: name<TAB>tailnet-fqdn<TAB>dump-storage-path
proxlab-gib       proxlab-gib.<your-tailnet>.ts.net       /var/lib/vz/dump
proxlab-bahamas   proxlab-bahamas.<your-tailnet>.ts.net   /var/lib/vz/dump
```

Three columns: friendly name shown in UI, Tailscale FQDN to SSH to, remote path where dump files land before restore.

Future v0.3: auto-discover peers by scanning tailnet for `tag:proxmox-host` devices via the Tailscale API. For v0.2, manual file is fine.

## Tailscale identity handling

When an LXC moves between hosts, what happens to its Tailscale identity?

### Option A: Fresh identity (default)

Before vzdump on source, wipe Tailscale state inside the container:
```bash
pct exec <vmid> -- bash -c '
  tailscale logout
  systemctl stop tailscaled
  rm -rf /var/lib/tailscale/*
'
```

Also re-arm the firstboot service so it'll join again on the destination:
```bash
pct exec <vmid> -- systemctl enable tailscale-firstboot.service
```

After restore on destination, Tupperware mints a new auth key and writes it to `/etc/tailscale/authkey`. On `pct start`, the firstboot service runs and the container joins the tailnet as a new device.

**User experience:**
- Tailnet IP changes (e.g., 100.10.20.30 → 100.40.50.60)
- Old entry shows as offline in admin console (eventually expires after 180d default)
- Hostname stays the same (Tailscale handles uniqueness — adds suffix if collision)

### Option B: Preserve identity (advanced)

Don't wipe Tailscale state. The vzdump captures `/var/lib/tailscale/tailscaled.state`, which contains the device's machine key. After restore, the container has the same identity.

**User experience:**
- Same tailnet IP
- Same admin console entry (just shows new direct/derp endpoints)
- No re-auth needed

**Risks:**
- If you accidentally start the source container after starting the destination, you have two nodes claiming the same machine key. Tailscale will refuse the second one (good) but it's confusing.
- If you ever restart the source for any reason post-transfer, it'll fight with the destination.

Mitigation: after a successful "preserve identity" transfer, Tupperware sets `pct set <vmid> -onboot 0` on the source to prevent accidental autostart. User still has to manually destroy.

### Implementation note

The choice is exposed as a radio button in the transfer dialog. Default fresh; preserve identity is the second option with a tooltip explaining the constraint.

## CLI tool

A new CLI command, used by the web UI but also runnable directly:

```bash
tupperware-transfer <local-vmid> <dest-name> [--preserve-identity]

# Examples
tupperware-transfer 200 proxlab-gib
tupperware-transfer 201 proxlab-bahamas --preserve-identity
```

`<dest-name>` matches a friendly name in `/etc/tupperware/peers.conf`. The script reads that file to get the FQDN and remote path.

Output is line-streamed (the web UI captures via Popen and streams via SSE).

## Error handling and recovery

### Source-side failures

| Failure point | Recovery |
|---|---|
| `pct stop` fails | Abort, container remains running, error to user |
| `vzdump` fails | Restart container (`pct start`), clean up partial file in `/tmp`, error to user |
| Local disk full during dump | Same as above |

### Network/transfer failures

| Failure point | Recovery |
|---|---|
| `scp` fails (network blip) | Restart source container, delete partial file on dest via SSH, error to user. **Important:** always clean up partial dump on dest, otherwise future restores might pick the wrong file. |
| Dest disk full | Same |

### Destination-side failures

| Failure point | Recovery |
|---|---|
| `pct restore` fails | Restart source container, clean up dump on dest, error to user |
| `pct start` on dest fails | Source remains stopped, dest container exists but won't start. Error to user with both VMIDs and instructions. |
| Tailscale firstboot fails on dest | Container is up, just not on tailnet. User can re-trigger via `pct exec <vmid> -- systemctl start tailscale-firstboot.service` after fixing the underlying issue (likely OAuth or DNS). |

### General principle

**Never destroy the source container automatically.** If anything goes wrong after the source is stopped, the worst case is "user has to `pct start` it again." Data is never lost.

If SCP completes but anything after fails, the dump file is preserved on both sides for manual recovery.

## Pre-flight checks (UI shows before user clicks "Start Transfer")

| Check | If fail |
|---|---|
| Source container exists | "VMID X not found" |
| Source container is running OR stopped (not template) | "VMID X is a template, can't transfer templates" |
| Destination is in peers.conf | "Destination not configured" |
| SSH to destination works | "Can't reach proxlab-gib via SSH (Tailscale ACL?)" |
| Source local disk has 2x container size free | "Need 8 GB free for dump, only 3 GB available" |
| Destination has Tupperware installed | "tupperware-receive not found on dest" |
| Destination has free VMID | "No free VMIDs on destination" |

UI should show these as a checklist that lights green/red before the "Start Transfer" button enables.

## Milestones

### Milestone 1: CLI plumbing

Just the bash tooling. No UI changes yet.

- `/etc/tupperware/peers.conf` format and parsing
- `/usr/local/sbin/tupperware-transfer` — the workhorse script
- `/usr/local/sbin/tupperware-rejoin` — helper for re-arming firstboot service after fresh-identity transfer (runs on dest via SSH)
- `/usr/local/sbin/tupperware-preflight` — pre-flight check script (returns JSON)

Test by running `tupperware-transfer` from CLI, verifying transfer works end-to-end.

### Milestone 2: Inventory tab in web UI

Read-only. Just lists LXCs with their stats. No transfer button yet.

- New `/inventory` route in Flask
- New tab navigation in HTML (NEW / TRANSFER)
- Refactor existing `INDEX` template to support tab structure

### Milestone 3: Transfer dialog

Wire up the inventory rows to the transfer logic.

- Per-row "Transfer" button
- Modal/expansion with destination dropdown + identity radio
- Pre-flight check display
- Live SSE streaming of `tupperware-transfer` output

### Milestone 4: Polish

- Error recovery testing (kill SCP mid-transfer, verify cleanup)
- Documentation in README + new `docs/transfer.md`
- Update screenshots
- Tag v0.2.0 release

## Open questions for v0.2

These don't block design but should be decided before milestone 4:

1. **Compression in vzdump**: zstd is default and fast. Do we want a flag to choose? Probably not — keep it simple, zstd always.
2. **Resume support for SCP**: rsync would give us resumable transfers. Worth the extra dependency? Probably not for v0.2; consider for v0.3.
3. **Storage backend on destination**: dest may use `local-lvm`, `local-zfs`, or directory storage. Today we'd need to specify `--storage` to `pct restore`. Add a column to peers.conf for restore-storage.
4. **Bandwidth limits**: large transfers over a slow link could saturate. Add a `--bwlimit` flag passable from the UI? Probably v0.3.
5. **Transfer audit log**: Should Tupperware keep a log of all transfers (when, what, where, by whom)? Useful for debugging and compliance. Probably yes, in `/var/log/tupperware/transfer.log`.

---

## Out of scope for v0.2

These are real features but explicitly not in this milestone:

- **Live migration** (memory state moves with running container)
- **Multi-host orchestration** (move 5 containers in one operation)
- **Authentication/authz on the web UI** (still tailnet-only access controls)
- **Web UI on destination shows incoming transfers in real time** — would need pub/sub or polling between Tupperware instances
- **Cross-version compatibility** (transferring between PVE 8 and PVE 9 hosts) — out of scope, both hosts must be same major version
